📐 2. ¿Es escalable la arquitectura?

✅
Sí escala como SaaS — el modelo de datos (negocio_id en todo) es el patrón estándar de multi-tenant de fila compartida. Funciona bien hasta miles de negocios en la misma DB.
⚠️
Límite actual: La DB es una sola instancia MySQL/MariaDB. Si llegás a cientos de negocios activos con alto tráfico, vas a necesitar separar en schemas por tenant o agregar caché (Redis). Hoy no es un problema.
💡
Para crecer: La estructura de archivos por módulo (api/restaurant/, api/gym/) permite agregar nuevos rubros sin tocar el código existente. Cada módulo es independiente.
📦
Cuando un cliente pide un cambio específico: si es visual → solo cambia la vista. Si es de lógica → solo cambia la API del módulo. Ninguno de los dos afecta al resto de los tenants.

🧩 4. Sistema de módulos por plan (lo que falta)

⚠️
El problema concreto hoy: La tabla planes tiene 5 flags genéricos (tiene_reportes, tiene_empleados...). No hay forma de controlar qué secciones específicas de cada rubro están incluidas en cada plan. Para hacer esto bien se necesitan dos tablas nuevas.

La solución — 3 tablas

-- 1. Cada sección del sistema tiene un slug único
CREATE TABLE modulos (
  id          INT PRIMARY KEY AUTO_INCREMENT,
  rubro_id    INT,                            -- NULL = módulo global
  clave       VARCHAR(60) UNIQUE NOT NULL,    -- ej: 'restaurant.reportes'
  nombre      VARCHAR(100) NOT NULL,          -- ej: 'Reportes Analytics'
  descripcion VARCHAR(200)
);

-- 2. Qué módulos tiene cada plan
CREATE TABLE plan_modulos (
  plan_id    INT NOT NULL,
  modulo_id  INT NOT NULL,
  PRIMARY KEY (plan_id, modulo_id)
);

-- 3. Overrides por negocio (para clientes especiales)
CREATE TABLE negocio_modulos (
  negocio_id INT NOT NULL,
  modulo_id  INT NOT NULL,
  activo     TINYINT(1) DEFAULT 1,   -- puede forzar ON u OFF
  PRIMARY KEY (negocio_id, modulo_id)
);

Cómo se usaría en runtime

// En cualquier API — una línea
PlanGuard::requireModule('restaurant.reportes', $negocioId, $db);

// En el sidebar — decide si mostrar el link
if (tieneModulo('restaurant.reportes')) { /* mostrar */ }

Lo que esto habilita desde el superadmin

Armar planes con checkboxes por módulo Dar acceso especial a un cliente sin cambiar su plan Bloquear un módulo a un cliente específico Agregar nuevos módulos sin tocar código de planes Controlar por rubro: "este plan de Restaurant incluye X, el de Gym incluye Y"

Estás a 3-4 semanas de un lanzamiento sólido. El trabajo restante son correcciones de seguridad, conectar PlanGuard, y pulir UX.
